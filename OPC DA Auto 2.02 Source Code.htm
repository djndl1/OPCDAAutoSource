Overview

Any use of the OPC DA Auto Wrapper Source Code is subject to the terms the OPC Foundation License Agreement.

The OPC DA Auto Wrapper Source Code consists of a ATL COM server that implements the Data Access Automation Interface 2.02 and two test client applications: one written with Visual Basic 6.0, the other written with Visual Basic .NET.

This code base was derived from the original automation wrapper code, however, it incorporates numerous bug fixes and design changes provided by various OPC Foundation members.

The DA Auto Wrapper and the .NET API

Both the DA Automation Wrapper and the OPC .NET API allow .NET based OPC client applications to communicate with existing COM based OPC servers. Users should keep in mind that the DA Automation Wrapper is intended to used only as a way to facilitate porting of existing VB 6.0 applications to VB .NET. The OPC Foundation recommends that any new VB .NET client development use the .NET API for the following reasons:

  • it provides almost all of the features of the DA automation wrapper;

  • it supports Data Access 3.00 and XML Data Access 1.00 servers;

  • it takes full advantage of new .NET features such as object serialization;

  • it requires one less layer of code between the application and the server;

  • it provides a common API framework for all specifications;

  • it will be the starting point for the Unified Architecture (UA) .NET API; 

This document includes a detailed mapping of the automation interface objects onto their equivalents in the .NET API.

System Requirements

Operating Systems

The DA Automation Wrapper is intended to run on all Windows systems that support DCOM, however, the OPC Foundation only tests the wrapper on Windows XP and Windows 2000. Vendors are asked to submit any fixes required for other platforms.

All source code provided by the OPC Foundation is only supported on Windows 2000 SP4 and Windows XP SP1. That said; the source code has been designed to be portable to other operating systems. For this reason vendors are encouraged to report fixes required to port the samples to other operating systems so they can be incorporated into the baseline and to improve the portability of future sample code.

All .NET components require that the .NET Framework Redistributable 1.1 or the .NET Framework SDK Version 1.1 is present on the target system. The .NET framework does not support Windows 95. All previously released OPC foundation binaries that require the .NET Framework 1.0 will run on systems with only the .NET Framework 1.1 installed. For this reason, users of OPC sample code should never need to install .NET Framework 1.0 on a machine.

Usage

VB .NET Memory Management

The .NET runtime environment manages COM object references differently from VB 6.0. Specifically, setting an object explicitly to Nothing will not release a COM server in VB .NET like it does in VB 6.0. As a result, users of the automation wrapper in VB .NET will likely find that COM servers are not being released when they expect. This will appear to be a reference count problem in the VB client, however, it may not be a concern since the .NET runtime environment is supposed to clean up the references automatically when the memory is 'needed'. Applications developers that do not wish to trust the .NET runtime environment may use the  Marshal.ReleaseComObject method to explicitly release COM servers.

OPCActivator

The default behavior of CoCreateInstanceEx is not suitable for some DCOM applications that have specialized security, licensing or network protocol requirements. As a result, the DA Automation wrapper includes an object called OPCActivator. This object allows VB applications to use vendor specific code to instantiate the DCOM server and then create a new instance of an OPCServer object that wraps an existing DCOM server instead of creating a new instance. 

Trace Log

The automation wrapper logs events to an HTML file in the working directory of the process that loads it. The log file name contain  executable name with '_OPCDAAuto_LOG.htm' appended to it. The content of these files is controlled with registry entries contained in the following key:

HKEY_LOCAL_MACHINE\SOFTWARE\OPC Foundation\OPCDAAuto\Trace \<Executable Name>

Value

Purpse

MaxEntries

The maximum number of entries that will be stored in the log (default=10000)

TraceLevel

A bit mask indicating what types of data will be stored in the log (default = 0x0000). 

The bits are:

 

0x0001 // any error messages.

0x0002 // additional detail about errors conditions.

0x0004 // any warning/informational messages.

0x0008 // additional detail about warning/informational conditions.

0x0010 // entry/exit from public functions.

0x0020 // before/after remote server calls.

0x0040 // in/out function parameters.

0x0080 // additional detail about in/out function parameters (e.g. array contents).

0x0100 // any other information.

0x0200 // additional detail about any other information.

Vendors are encouraged to report problems, bug fixes, enhancements and any other comments about the components supplied by the OPC Foundation. To support this effort, the OPC Foundation has technical staff prepared to review and incorporate feedback into the baseline and release new versions of the components and sample code in a timely manner for the benefit of all members.

Please note that this support is intended for OPC Foundation members who are incorporating OPC support into their products.
